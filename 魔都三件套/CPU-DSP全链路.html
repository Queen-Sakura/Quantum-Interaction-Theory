F0 · 地基 异构计算 · 两块处理器在同一块硅上 85/100
物理事实第一条
CPU 和 DSP 是两块独立处理器，共享 DRAM，不共享缓存
┌─────────────────────────────────────────────────────┐ │ Snapdragon SoC (同一块硅) │ │ │ │ ┌───────────────────┐ ┌────────────────────────┐ │ │ │ Kryo CPU (ARM) │ │ Hexagon DSP │ │ │ │ L1$ L2$ L3$ │ │ L1$ L2$ │ │ │ │ MMU (页表) │ │ (无完整 MMU, 靠 SMMU) │ │ │ │ 跑 Linux/Android │ │ 跑 QuRT (RTOS) │ │ │ └───────┬───────────┘ └───────────┬────────────┘ │ │ │ │ │ │ │ ┌──────┬──────────┐ │ │ │ └────┤共享 DRAM│ 邮箱中断 ├────┘ │ │ └──────┴──────────┘ │ └─────────────────────────────────────────────────────┘
⏳ 为什么不是 SMP?
CPU 和 DSP 是 异构处理器——指令集不同、缓存独立、MMU 独立、跑不同 OS。通信和两台网络连接的电脑一样——只是"网线"换成了共享内存总线。
1F · 物理层 共享内存 + 门铃 = 两条通道 88/100

两条物理通道

通道物理介质方向延迟
数据通道SMEM — DRAM 中保留区域双向读写~100ns
信号通道硬件邮箱中断 — 硅上触发器CPU→DSP / DSP→CPU~μs
物理事实第二条
共享内存只是"写了一块物理地址"。对方怎么知道你写了？——门铃中断。

为什么不能轮询？ 省电。DSP 大部分时间在 idle（clock gating），轮询 = CPU 一直读 DRAM = 功耗爆炸。中断唤醒后才读。

门铃物理路径：CPU 写硬件寄存器 → 硅触发器翻转 → IRQ 线电平变化 → DSP GIC 比较器 → 拉 DSP 核 IRQ。和上次审计的 GPIO→GIC→CPU 完全同构。

2F · SMEM 共享内存分配器 · 两国外交部都认的地 80/100

SMEM 不是 Linux 的一部分。 高通在系统启动时从 DRAM carve out 保留区域。地址 DT 硬编码，CPU MMU 和 DSP SMMU 都配了映射。

为什么 malloc 不能替代 SMEM？

  • Linux buddy allocator 管 CPU 虚拟地址 → 物理地址
  • DSP 不认 CPU 页表
  • SMEM 物理地址在启动阶段就已约定——不需要运行时协商
⏳ 狮山成因
SMEM = 两个主权国家之间的中立瑞士。不需要互认领土主权（MMU），只认一块共同约定的物理地块。
3F · GLINK 传输层 · FIFO + 中断 + Intent 流控 75/100

GLINK 做四件事

职责怎么做如果没有
封装RPMsg 消息 → GLINK 协议包 (+CMD +intentID +CRC)裸二进制，分不清边界
FIFO 管理共享内存环形缓冲区 + head/tail 指针写覆盖读
中断触发写完调 mbox_send_message() 踢对面对面不知道
Intent 流控接收方预分配 buffer 并"宣布"——发送方只能往已宣布 intent 里写溢出

Intent = 背压机制

CPU 发 4KB → 查 DSP intent 列表 → 选 intent#3(4KB) → 标记占用 → 写 SMEM → DSP 处理 → RX_DONE → 归还 intent#3

为什么是 Intent 而不是直接写+丢弃？ DSP 没有虚拟内存，没有 swap。溢出 = 丢数据 = 不可接受。Intent = 内核 ring buffer 同构——生产者-消费者，预分配，指针推进。

变体在哪用于
GLINK-RPM片上 SRAM电源管理
GLINK-SMEMDRAM SMEMDSP/ADSP/Modem → FastRPC
⏳ 狮山成因
Intent 机制决定了 FastRPC 的数据路径：不能发 > intent 大小的消息 → 必须拆包或走 dma-buf 零拷贝。fastrpc.c 的 inline 和 mmap 两条路径由此分流。
4F · RPMsg 消息协议 · 多信道复用 85/100

RPMsg = GLINK 之上的寻址层。 GLINK 只负责可靠传输 blob。同一条物理信道上跑 FastRPC、音频 APR、传感器——靠 RPMsg dst 字段区分。

消息头：[src(32b)] [dst(32b)] [len(16b)] [flags(16b)] + payload

类比：GLINK = IP。RPMsg = UDP 端口号。FastRPC = HTTP。

⏳ 知识边界
flags 位定义、reserved 字段、错误处理不影响理解 FastRPC。已达标。
5F · remoteproc DSP 固件启动 · 开机才有通信 75/100

remoteproc 不是通信层——是通信的前提。

FastRPC 可用前 DSP 必须：开机 → 加载固件 → 初始化 QuRT → 注册 GLINK 信道 → 启动 dispatcher。

  1. 从 /lib/firmware/ 读 ELF
  2. 解析 segment → 载入 DSP 可访问内存
  3. 释放 reset → DSP 开始执行
  4. 崩溃时重新加载

6F · SMMU IOMMU 上下文银行 · 安全核心 50/100 ⚠️

为什么 DSP 需要 SMMU？

DSP 能直接访问物理内存。没有隔离 = 用户 A 可传指针让 DSP 读用户 B 内存。SMMU 限制 DSP 只能访问被明确映射的物理页。

Context Bank = 独立页表

CPU 进程 A CPU 进程 B │ │ ▼ ▼ CB #1 (ctx bank 1) CB #2 (ctx bank 2) IOVA → PA_A IOVA → PA_B │ │ ▼ ▼ DSP 通过 CB#1 只能读 PA_A DSP 通过 CB#2 只能读 PA_B

保护域 (PD) = 一个 CB + 一套页表。 DSP 侧加载器在 mmap ELF segment 前先检查 PD 签名权限。

Signed PDUnsigned PD
签名必须 OEM/QC不要求
节点/dev/fastrpc-*-secure/dev/fastrpc-*
支持全部 DSP仅 CDSP, GDSP

9F · Cache Coherency ⚠️ 实战 bug 第一来源 · 四 flag 围着一个物理问题 40/100

物理根源

CPU L1/L2 Cache DSP L1/L2 Cache │ │ └────────┬───────────────────┘ │ (两条独立 SRAM — 无硬件 coherency 协议) ▼ DRAM (CPU 写了但没 flush → DSP 读到旧的 DSP 写了但没 invalidate → CPU 读到旧的)

方向性：

  • CPU 写 → DSP 读：CPU 必须 flush（写回 DRAM），DSP 必须 invalidate（丢弃缓存行）
  • DSP 写 → CPU 读：DSP 必须 flush，CPU 必须 invalidate

四个 ATTR Flag

Flag效果何时用
COHERENT映射为 uncached / SMMU coherent小 buffer，频繁读写，不在乎延迟
NON_COHERENT映射为 cached，手动维护大数据量，cache 友好访问模式
FORCE_NOFLUSH跳过 CPU flush确定 CPU 已 flush 过
FORCE_NOINVALIDATE跳过 CPU invalidate确定 CPU 没读过这片
⚠️ 最关键待深入 (40/100)
精确的 cache 维护操作序列、与 SMMU 的交互、COHERENT 映射到底做了什么——需要读内核代码确认。这是 Argo 在审的代码里极可能遇到的关键路径。
