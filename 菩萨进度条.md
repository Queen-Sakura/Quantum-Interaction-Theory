## 第十八波 · 楚国内核驱动全链路贯通（7.2）

### 广度优先 · 认知方法论

元首发现：搞 Linux 应广度优先，非深度优先。先看清全图——每一层是什么、为什么存在、层和层之间怎么握手。具体函数的参数表、寄存器的位定义——那是手册的事，不是认知的事。内核是横跨硬件和用户态的桥梁——先理解桥的结构、承重、连接点，不必先算每颗铆钉的应力。

### 全链路贯通 · 从硬件到 /dev

**设备号注册**：`register_chrdev(0, name, &fops)` 是 Linux 标准动作。告诉内核三件事——我叫什么、这四个函数是我的、给我分身份证号。`mknod /dev/hello c <major> <minor>` 在文件系统贴标签。两件事缺一不可：去公安局登记户口 + 在门口挂门牌号。

**启动顺序**：内核用 initcall 分等级——pure→core→postcore→arch→subsys→fs→device→late。你的驱动 `module_init()` 落在 device_initcall (6)。PCI 总线在 subsys_initcall (4) 先起来。依赖链自动满足。B 依赖 A → `MODULE_SOFTDEP("pre: A")`。

**PCIe 驱动匹配**：不是你在代码里找设备。内核枚举总线 → 读 Vendor ID / Device ID → 问所有登记过的驱动"谁认识这个？"→ 你的 `id_table` 匹配 → `probe()` 被调。不是调用内核——内核调用你。回调。

**probe 六件事**：上电(`pci_enable_device`)→ 映射寄存器(`pci_iomap`)→ 分配私有结构 → 注册中断 → 创建 /dev 节点 → `pci_set_drvdata`。本质：把一块硅变成文件。

**两级供电**：插槽 = Vaux 辅助电源（够读配置空间·枚举）。`pci_enable_device` = 开 Vcc 功能电源（寄存器读写·中断·功能部分）。物理上插了卡 ≠ 功能上电。

**MMIO 映射**：`ioread32(base + offset)` → CPU 执行 load → MMU 标记 uncached → 北桥识别地址为 PCIe MMIO 窗口 → 路由到 PCIe 总线 → TLP 包 → 卡解包→ 返回数据。不是内存。不是缓存。全链路 PCIe 协议保证。

### DSP 驱动 · 提线木偶

CPU 内核驱动 = 操偶师的手。DSP = 木偶。线 = 共享内存 + 中断。probe 开机建线。read/write/ioctl 拉线。

**Glink** = 高通 CPU-DSP 传输层。只负责二进制搬运——不解析。信道名对上 = 连接建立。包格式、命令号、CRC——两边用同一个 struct 定义。

**德国银行模型**：填 Glink 包 → `glink_write` → `wait_for_completion_timeout` → DSP 回复 → 中断处理函数 `complete(&done)` → 唤醒 → `copy_to_user`。看门狗 = `schedule_delayed_work`——5 秒不回 → 复位/重试/重启信道。

**内核 = 路由表**：Glink 连着 RPM(电源)、Modem(基带)、ADSP(音频)、SLPI(传感器)、CDSP(计算)、WCN(WiFi)、Venus(视频)、GPU。内核驱动只暴露该暴露的信道。用户态 `cat /dev/glink_raw` → `write("rpm_request", "CPU 电压调到 0V")` → 系统断电——内核不让你填其他表。

### 中断 · 物理全链路

DSP 回包 → 写 doorbell 寄存器（硅上触发器翻转）→ GPIO-IRQ 线电平变化 → GIC 物理比较器检测 → 拉 CPU IRQ 物理线 → CPU 完成当前指令 → 保存上下文 → 跳转中断向量表 → `irq_handler()`。每一层都是物理的。

**上半部**：中断上下文。几微秒。只确认中断源 + 搬数据 + 排下半部。**下半部**：tasklet/workqueue。可调度上下文。可被打断。真正的协议解析。

GIC 优先队列：优先级高的插队。同优先级排队。下半部不抢占上半部。接口频率不高 → 不存在风暴。

### DMA 发现

元首独立发现 DMA 原理："把数据搬运交给单独器件不行吗？为什么要 CPU 做？"——对。DMA 控制器接管总线，CPU 继续跑。搬完才通知 CPU。CPU 从头到尾没碰数据。ARM SoC 没有为每个中断源做 DMA 触发通路——是工程折中，不是物理定律。

### 广度优先 · 封板标准

内核认知框架完整：设备注册→总线匹配→probe 初始化→中断处理→用户态接口。每一层的物理约束、协议保证、为什么这样而非那样。足够。更深细节 = 手册 + 具体芯片 datasheet。

### 全链路贯穿 · ioctl 路由本质

元首贯通全链路：用户态 `invoke()` → SDK → ioctl(命令号) → VFS 路由到 fops → switch(cmd) 跳到驱动函数 → Glink 发包 → 共享内存 + 门铃 → GPIO 电平变化 → GIC 物理比较器 → CPU IRQ 被拉高 → 中断向量表 → DSP 固件。

**ioctl 的本质**：一个数字 → 跳到你的一个函数。和中断向量表同构——中断号 → 中断处理函数。VFS、ioctl switch、Glink 信道名——全是查表跳转。全链路同一模式：路由 → 执行 → 返回。

**ioctl 做了四件事**：路由(cmd 决定跳到哪个 case) + 类型检查(幻数·命令号·参数大小) + 权限校验(fd 权限) + 拷贝(copy_from_user)。拷贝是最后一件。

**/dev/dsp0 唯一性**：全系统唯一。任何用户打开它 → VFS 解析 inode → `(major, minor)` → 同一个 `struct cdev` → 同一个 `struct file_operations`。设备唯一，但每个 fd 的 `private_data` 可独立。

**SDK 的角色**：用户不需要知道 ioctl。Hexagon SDK 调 ioctl。用户只写 `remote_handle.invoke(method_id, &args)`。一行。

从"register_chrdev 是什么"到"全链路贯穿"——一个 session。不是背手册——是每层推到物理约束，从物理推出为什么只有这条路。广度优先的工程学验证：先看清全图，每一层为什么存在、层和层之间怎么握手。框架在脑子里，细节查手册即可。
